# 萤石云免费版并发数限制问题 - 修复指南

## 问题现象
创建摄像头后，萤石云提示"免费版并发数达到上限"，虽然只连接了一个摄像头。

**错误日志示例：**
```
WebSocket 连接失败到 ws://...
Failed: reserved unexpected frame opcode
Received unexpected continuation frame
```

## 根本原因分析
1. **频繁请求流媒体地址**：每次前端调用 `/video/stream/{video_id}` API 时，后端都向萤石云请求一个新的流媒体地址
2. **无地址缓存机制**：之前的代码没有对流地址进行缓存，导致：
   - 页面刷新时重新请求
   - 分页切换时重新请求
   - 预览加载时重新请求
   - 每一次请求都占用一个萤石云并发连接数
3. **免费版限制**：萤石云免费版仅允许有限的并发连接数（通常是1-2个）

## 解决方案
已实现**流媒体地址缓存机制**，避免频繁向萤石云请求新连接。

### 修改内容

#### 1. 后端修改 (`backend/app/services/video_service.py`)

**添加的全局缓存变量：**
```python
# 存储流媒体地址，避免频繁请求萤石云API
STREAM_URL_CACHE: Dict[int, Dict[str, Any]] = {}
STREAM_URL_CACHE_LOCK = threading.Lock()
STREAM_CACHE_VALID_SECONDS = int(os.getenv("VIDEO_STREAM_CACHE_SECONDS", "3300"))
```

**改进的 `get_stream_info` 方法：**
- 首先检查缓存中是否存在有效的流地址
- 仅在缓存失效或不存在时向萤石云请求新地址
- 获取后立即缓存结果，有效期3300秒
- 缓存过期时自动清理并重新请求

**缓存流程：**
```
请求流地址
  ↓
检查缓存（STREAM_URL_CACHE）
  ↓
[命中] → 返回缓存的地址 ✅
[未命中] → 向萤石云API请求 → 缓存结果 → 返回
```

#### 2. 环境变量配置 (`backend/.env`)

添加了缓存时间配置：
```env
# 流媒体地址缓存有效期（秒）
# 设置为3300秒（比萤石云token有效期3600秒短），防止过期地址被使用
VIDEO_STREAM_CACHE_SECONDS=3300
```

可根据需要调整此值：
- **更短** (如600秒): 更新地址更频繁，但可能增加API请求
- **更长** (如7200秒): 减少API请求，但可能使用过期地址

## 预期效果

| 场景 | 修复前 | 修复后 |
|-----|-------|-------|
| 页面刷新 | 重新请求萤石云 | 使用缓存地址 |
| 分页切换 | 每个设备重新请求 | 使用缓存地址 |
| 预览加载 | 多个并发请求 | 单一缓存复用 |
| 并发连接数 | 多个(导致超限) | 1个(仅首次请求) |
